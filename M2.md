# Milestone 2 — CRAGB v1 Construction (E1)

**Depends on:** M1 (corpus_v1 + EDA, complete).
**Hardware note:** built on a laptop RTX 3050 (4–6 GB VRAM class, shared with OS/display) — lighter than the RTX 4060/8GB assumed in PLAN.md §1.3. This affects T2.5 (dense retriever): stick to small embedding models (`bge-small-en-v1.5` / `all-MiniLM-L6-v2`, both <150M params, run comfortably even on CPU) and keep the FAISS index **flat/exact** as already planned (§12) — no ANN tuning needed at this corpus size regardless of GPU. Embed in batches (e.g. 64–128) rather than encoding the whole corpus in one call to avoid VRAM spikes; CPU fallback is acceptable if VRAM is tight since encoding hundreds of thousands of short review chunks with a small model is not the bottleneck (§1.4 risk G — the *fine-tuning* stage is where 8GB-class VRAM actually bites, not retrieval).

Per the dependency graph in PLAN.md §2/§4, M2 needs a minimal retriever to run the **pooling protocol** — so retriever build is included here as an explicit sub-task rather than assumed pre-existing. If BM25+dense already exist from M1, skip T2.4/T2.5.

---

### T2.1 — Define question taxonomy
- **Artifact:** `benchmark/taxonomy.md`
- **Output:** 7 categories (fit/sizing, colour/appearance, fabric/quality, durability, defects, occasion, value) with target question counts per type, including a deliberate negative/unanswerable quota (per H4/risk C).
- **Validation:** counts sum to ~60; every category has ≥1 negative-style question; taxonomy reviewed against corpus_v1 EDA (does the corpus actually have enough defect/negative reviews to support it?).
- **Commit:** `feat(bench): define CRAGB question taxonomy`
- **Mid-progress report:** Yes — feeds the "corpus/benchmark composition" table (§7).

### T2.2 — Draft candidate questions (LLM-draft → human curate)
- **Artifact:** `benchmark/cragb_draft.jsonl`
- **Output:** ~80–100 candidate questions generated from sampled reviews (over-generate to allow curation losses), tagged by taxonomy type, with a versioned generation prompt saved to `src/cragb/generate/prompts/benchmark_gen.md`.
- **Validation:** spot-check 10 questions aren't trivially answerable from title/metadata alone (must require review text); no duplicate/near-duplicate questions; taxonomy balance close to T2.1 targets.
- **Commit:** `feat(bench): LLM-drafted candidate questions`
- **Mid-progress report:** Yes — the prompt is one of the required "versioned prompts" (§7).

### T2.3 — Curate to final ~60 questions
- **Artifact:** `benchmark/cragb_questions_v1.jsonl` (question, type, id — no labels yet)
- **Output:** human-curated final set, rejecting bad drafts, rebalancing taxonomy, ensuring the image-bearing subset is targeted (needed for RQ4/E7 later).
- **Validation:** final count ≈60 ±5; taxonomy distribution matches T2.1 targets within reason; each question is answerable in principle from the corpus (manually verify a handful against corpus_v1).
- **Commit:** `feat(bench): curate final CRAGB question set`
- **Mid-progress report:** Yes — this table is required for the report.

### T2.4 — Minimal `Retriever` interface + BM25 index (if not already built)
- **Artifact:** `src/cragb/retrieval/base.py`, `src/cragb/retrieval/bm25.py`
- **Output:** `Retriever.index()` / `.search(q, k)` interface; BM25 built over `corpus_v1` chunks.
- **Validation:** unit test in `tests/test_retrieval.py` — known query returns expected top document on a small synthetic fixture; smoke test on 3 real CRAGB questions returns non-empty results.
- **Commit:** `feat(retrieval): BM25 retriever + interface`
- **Mid-progress report:** Partial — underlying infra for RQ2 figures, not itself a report artifact.

### T2.5 — Dense retriever (bge-small / MiniLM) + FAISS flat index
- **Artifact:** `src/cragb/retrieval/dense.py`
- **Output:** embedding index over `corpus_v1`, same interface as BM25. Small model, batched encoding (see hardware note above) — CPU or 3050 both fine at this scale.
- **Validation:** unit test confirms cosine ranking is sane on a fixture; index build time/size logged (feeds E6 later); same smoke test as T2.4 run through dense retriever.
- **Commit:** `feat(retrieval): dense retriever (bge-small) + FAISS flat index`
- **Mid-progress report:** Partial — same as T2.4.

### T2.6 — Run pooling: union of top-k across retrievers
- **Artifact:** `benchmark/pools_v1.jsonl` (question_id → pooled candidate doc_ids)
- **Output:** for each of the ~60 questions, run BM25 + dense at k∈{5,10}, take the union → candidate pool per question.
- **Validation:** every question has a non-empty pool; log pool size distribution (flag any question with a suspiciously tiny pool — may need k raised or question rewritten).
- **Commit:** `feat(bench): pooled retrieval candidates for CRAGB`
- **Mid-progress report:** No (intermediate artifact) — but the pool-size stats are worth one sentence in the datasheet.

### T2.7 — Human relevance labeling over pools
- **Artifact:** `benchmark/relevance_labels_v1.jsonl` (question_id, doc_id, relevant: 0/1)
- **Output:** binary-label every pooled candidate per question — the scarce-resource step flagged as a bottleneck in PLAN.md §1.4.
- **Validation:** every pool has ≥1 positive label (else question is unanswerable and should be flagged as a deliberate negative or dropped); log time spent for the datasheet.
- **Commit:** `feat(bench): human relevance labels over pooled candidates`
- **Mid-progress report:** Yes — "% questions with ≥1 relevant doc" is a required stat (§3-E1).

### T2.8 — Write reference answers from labeled-relevant reviews
- **Artifact:** `benchmark/reference_answers_v1.jsonl`
- **Output:** human-written grounded answer per question, composed only from reviews labeled relevant in T2.7 (or "not enough information" for deliberate negatives).
- **Validation:** spot-check each answer cites only relevant doc_ids; negatives correctly produce an abstention-style reference.
- **Commit:** `feat(bench): human reference answers for CRAGB`
- **Mid-progress report:** Yes — needed for RQ0/RQ1 similarity scoring (E5).

### T2.9 — Select best-evidence photo per image-bearing question
- **Artifact:** `benchmark/best_photo_v1.jsonl`
- **Output:** for questions where relevant reviews include images, pick the single most probative photo.
- **Validation:** coverage % computed (how many questions got a usable photo) — must be reported honestly per H4/risk D, not inflated.
- **Commit:** `feat(bench): best-evidence photo selection`
- **Mid-progress report:** Yes — feeds the RQ4/multimodal coverage figure.

### T2.10 — Assemble `cragb_v1.jsonl` + held-out split + datasheet
- **Artifact:** `benchmark/cragb_v1.jsonl`, `benchmark/cragb_v1_datasheet.md`
- **Output:** merge T2.3/T2.7/T2.8/T2.9 into the final schema (question, type, relevant_ids[], reference_answer, best_photo_id); mark whole set as held-out (never used for fine-tuning) with a hash guard for the T2.8 leakage check later (E8).
- **Validation:** schema check (all required fields present, no null relevant_ids for non-negative questions); run a leakage hash test stub in `tests/test_no_leakage.py`.
- **Commit:** `feat(bench): assemble CRAGB v1 + datasheet + leakage guard`
- **Mid-progress report:** Yes — this is *the* benchmark deliverable, headline artifact of the whole milestone.

**Sequencing:** T2.1 → T2.2 → T2.3 (can start in parallel with T2.4/T2.5) → T2.4 + T2.5 → T2.6 → T2.7 → T2.8 + T2.9 (parallel) → T2.10.

**Total:** ~10 tasks, roughly matching the 1.5-week M2 estimate in PLAN.md §4 (assuming T2.4/T2.5 aren't already done from M1, in which case cut to ~1 week).
