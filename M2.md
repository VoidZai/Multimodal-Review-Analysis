# Milestone 2 — CRAGB v1 Construction (E1)

**Depends on:** M1 (corpus_v1 + EDA, complete).
**Hardware note:** built on a laptop RTX 3050 (4–6 GB VRAM class, shared with OS/display) — lighter than the RTX 4060/8GB assumed in PLAN.md §1.3. This affects T2.5 (dense retriever): stick to small embedding models (`bge-small-en-v1.5` / `all-MiniLM-L6-v2`, both <150M params, run comfortably even on CPU) and keep the FAISS index **flat/exact** as already planned (§12) — no ANN tuning needed at this corpus size regardless of GPU. Embed in batches (e.g. 64–128) rather than encoding the whole corpus in one call to avoid VRAM spikes; CPU fallback is acceptable if VRAM is tight since encoding hundreds of thousands of short review chunks with a small model is not the bottleneck (§1.4 risk G — the *fine-tuning* stage is where 8GB-class VRAM actually bites, not retrieval).

Per the dependency graph in PLAN.md §2/§4, M2 needs a minimal retriever to run the **pooling protocol** — so retriever build is included here as an explicit sub-task rather than assumed pre-existing. If BM25+dense already exist from M1, skip T2.4/T2.5.

### T2.1 — Define question taxonomy
- **Objective:** Fix the 7 CRAGB question categories (fit/sizing, colour/appearance, fabric/quality, durability, defects, occasion, value) with target counts and a deliberate negative/unanswerable quota per category (per H4/risk C), and confirm `corpus_v1` actually supports each category before any question is drafted.
- **Files to create:**
  - `configs/taxonomy.yaml`
  - `src/cragb/bench/__init__.py`, `src/cragb/bench/taxonomy.py`
  - `tests/test_taxonomy.py`
  - `benchmark/taxonomy.md`
- **Python packages:** `pandas`, `pyyaml`, `pytest`
- **Expected outputs:** `benchmark/taxonomy.md` — a table of the 7 categories with target/negative counts and each category's corpus keyword-coverage %, plus a PASS/FAIL validation block.
- **How I verify it worked:** `pytest tests/test_taxonomy.py` all green; running the module against real `corpus_v1.parquet` prints `PASS` and every category clears the coverage floor (no near-zero category); counts sum to ~60 with every category having ≥1 negative-style question.
- **Git commit:** `feat(bench): define CRAGB question taxonomy`
- **Mid-progress report:** Yes — feeds the "corpus/benchmark composition" table (§7).

### T2.6 — Run pooling: union of top-k across retrievers
- **Objective:** Generate the candidate document pool per question by taking the union of BM25 and dense top-k results — the TREC pooling remedy for incomplete relevance judgments (risk A).
- **Files to create:**
  - `src/cragb/bench/pooling.py`
  - `benchmark/pools_v1.jsonl` (question_id → pooled candidate doc_ids)
- **Python packages:** `pandas`, `json` (stdlib)
- **Expected outputs:** for each of the ~60 questions, the union of BM25 + dense top-k (k∈{5,10}) candidate doc_ids.
- **How I verify it worked:** every question has a non-empty pool; pool-size distribution logged, with any suspiciously tiny pool flagged (may need k raised or the question rewritten).
- **Git commit:** `feat(bench): pooled retrieval candidates for CRAGB`
- **Mid-progress report:** No (intermediate artifact) — but the pool-size stats are worth one sentence in the datasheet.

### T2.7 — Human relevance labeling over pools
- **Objective:** Binary-label every pooled candidate per question — the scarce-resource step flagged as a bottleneck in PLAN.md §1.4 — so Recall/Hit@k are computed against trustworthy, TREC-style-complete judgments.
- **Files to create:**
  - `src/cragb/bench/label_relevance.py` (helper to iterate pools and record labels)
  - `benchmark/relevance_labels_v1.jsonl` (question_id, doc_id, relevant: 0/1)
- **Python packages:** `pandas`, `json` (stdlib)
- **Expected outputs:** a relevance label for every (question, doc_id) pair in every pool from T2.6.
- **How I verify it worked:** every pool has ≥1 positive label (else the question is unanswerable and should be flagged as a deliberate negative or dropped); time spent logged for the datasheet.
- **Git commit:** `feat(bench): human relevance labels over pooled candidates`
- **Mid-progress report:** Yes — "% questions with ≥1 relevant doc" is a required stat (§3-E1).
